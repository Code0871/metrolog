## технический стек:

#### Библиотеки обработки данных:
- **Pandas** — основная библиотека для манипуляции табличными данными. Обеспечивает чтение данных из PostgreSQL, очистку, фильтрацию и агрегацию. Выбрана как стандарт де-факто в анализе данных на Python
- **NumPy** — фундамент для векторных вычислений. Используется как основа для TensorFlow и CatBoost, а также для вычисления пользовательских метрик (PSI, custom loss)
- **Dask** — фреймворк для отложенных (lazy) вычислений на данных, не помещающихся в память. Парк СИ предприятия содержит данные за 10+ лет с сотнями тысяч записей — pandas при полной загрузке вызывал падение по памяти (OOM). Dask позволил обрабатывать полный объём исторических данных, не жертвуя pandas-совместимым API

**Обоснование выбора:** связка pandas + Dask даёт масштабируемость без переписывания кода. Альтернативы (PySpark, Polars) требовали бы полной миграции пайплайнов

#### Модели машинного обучения:
- **CatBoost** — градиентный бустинг Яндекса, оптимизированный под табличные данные с категориальными признаками. Парк СИ содержит множество категорий (тип СИ, производитель, подразделение) — CatBoost обрабатывает их нативно, без one-hot encoding. Дополнительно используется встроенный SHAP для объяснимости прогнозов закупок
- **TensorFlow** — фреймворк глубокого обучения. Применяется для обучения модели классификации СИ по степени износа на основе многомерных временных рядов (история поверок, ремонтов). Выбран потому, что позволяет строить гибкие архитектуры (LSTM, attention) под нестандартную структуру последовательностей метрологических событий

**Обоснование выбора:** CatBoost — лучший выбор для задачи регрессии закупок (табличные данные, высокое качество из коробки). TensorFlow оправдан для задачи классификации на последовательностях, где градиентный бустинг уступает глубоким архитектурам. Разделение инструментов по задачам дало суммарный прирост точности ~7% против единой модели

#### Хранилища данных:
- **PostgreSQL** — основное реляционное хранилище, источник истины (source of truth). Хранит нормализованные данные: реестр СИ, историю поверок, закупки, справочники. Выбран за надёжность, поддержку JSONB для гибких метаданных и полнотекстовый поиск `tsvector` — последний используется как baseline для сравнения с AI-поиском
- **Qdrant** — векторная база данных, оптимизированная под approximate nearest neighbor (ANN) поиск. Хранит эмбеддинги описаний СИ для семантического поиска. Позволяет находить «манометр» по запросу «датчик давления», что недоступно `LIKE` и ограниченно доступно `tsvector`. Выбрана среди аналогов (Milvus, Weaviate) за минимальное потребление ресурсов и нативную Rust-реализацию с Python API
- **Redis** — in-memory key-value хранилище для кеширования. Кеширует результаты прогнозов классификации (TTL: 1 сутки) и частые поисковые запросы (TTL: 1 час). Снизил медианное время ответа API с 1.8 сек до 180 мс на повторных обращениях. Выбран как самый лёгкий и предсказуемый по latency инструмент

**Обоснование выбора:** три хранилища решают три разные задачи, не конфликтуя. PostgreSQL — надёжность и целостность, Qdrant — семантический поиск, Redis — скорость. Попытка решить всё одной БД привела бы к компромиссам по каждому измерению CAP-теоремы

#### Инфраструктура вокруг модели:
- **Python** — язык разработки моделей, пайплайнов обработки данных и обучения. Выбран за экосистему: pandas, CatBoost, TensorFlow, MLflow не имеют полноценных аналогов в других языках. Инференс моделей также выполняется на Python в отдельных микросервисах
- **Go** — язык API-сервисов и шлюза. Выбран для высоконагруженного слоя, обслуживающего поисковые запросы и прогнозы. Ключевые преимущества: компиляция в нативный код, горутины для конкурентной обработки (обработка 500+ RPS поиска без деградации), низкое потребление памяти. Python-микросервисы с моделями вызываются через gRPC
- **MLflow** — платформа для трекинга экспериментов и версионирования моделей. В проекте залогировано 12+ экспериментов с CatBoost (подбор гиперпараметров, feature engineering) и 5 экспериментов с TensorFlow (архитектуры LSTM vs Transformer). MLflow Model Registry обеспечивает staged-продвижение: Staging → Production → Archived с фиксацией метрик. Выбран за открытость, низкий порог входа и интеграцию с Python-стеком «из коробки»

**Обоснование выбора языков:** Python — для того, что требует научной экосистемы; Go — для того, что требует производительности и надёжности. Связка через gRPC даёт чёткий контракт и независимое масштабирование. MLflow выбран вместо коммерческих аналогов (Weights & Biases, Neptune) для сохранения полного контроля над данными экспериментов внутри контура предприятия
